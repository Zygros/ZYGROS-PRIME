### 1. Set Up the Arrakis Server

First, you need to host the Arrakis service on your own infrastructure, which must be a Linux machine with virtualization enabled.

*   **Prerequisites**: Arrakis uses `cloud-hypervisor`, which requires `/dev/kvm` on a Linux machine. You can check if virtualization is enabled by running `stat /dev/kvm`.
*   **Quick Setup**: The easiest way to get started is by using the provided `setup.sh` script, which downloads all the necessary pre-built binaries and images.
    ```bash
    curl -sSL https://raw.githubusercontent.com/abshkbh/arrakis/main/setup/setup.sh | bash
    ```
*   **Run the Server**: After setup, you need to run the `arrakis-restserver`. This server manages the lifecycle of the sandboxes (MicroVMs).
    ```bash
    cd arrakis-prebuilt
    ./arrakis-restserver
    ```
This server exposes a REST API that your AI agent will communicate with.

### 2. Use the Python SDK (`py-arrakis`) in Your Agent's Code

Arrakis provides a Python SDK called `py-arrakis` to programmatically control the sandboxes. This is the primary way your AI agent will interact with the sandboxed environment.

Here is a Python code example from the documentation demonstrating how to use the SDK:

```python
from py_arrakis.sandbox_manager import SandboxManager

# Connect to the Arrakis server you are hosting.
# Replace with the IP and port where arrakis-restserver is running.
sandbox_manager = SandboxManager('http://127.0.0.1:7000')

# Start a new sandbox.
# The 'with' statement ensures the sandbox is automatically destroyed after use.
with sandbox_manager.start_sandbox('my-agent-sandbox') as sb:
    # Now you can execute commands inside the isolated sandbox
    result = sb.run_cmd('echo "Hello from inside the sandbox!"')
    print(result['output'])

    # You can also upload and download files, run complex scripts, etc.
```

### 3. Key Integration Concepts

Your AI agent's logic will use the SDK to perform actions within the sandbox. Here are the core functionalities you can integrate:

*   **Executing Commands**: Use `sandbox.run_cmd()` to run any shell command inside the Ubuntu-based VM. The agent can then parse the `stdout` and `stderr` from the command's output to understand the result and decide the next step.
*   **File Management**: Your agent can write files into the sandbox (e.g., source code, scripts) and read files out of it (e.g., program output, log files).
*   **Snapshot and Restore**: This is a powerful feature for complex tasks. If an agent is following a multi-step plan, it can create a snapshot of the VM's state. If a subsequent step fails, it can restore the VM to the last known good state and try a different approach.

Here's an example of the snapshot and restore workflow:

```python
# ... inside the 'with' block ...
sb.run_cmd("echo 'Initial state' > /tmp/progress.txt")

# Create a snapshot of the current state
snapshot_id = sb.snapshot("checkpoint-1")
print(f"Created snapshot with ID: {snapshot_id}")

sb.run_cmd("echo 'This might fail' >> /tmp/progress.txt")

# Let's say the agent decides to backtrack. First, destroy the current sandbox.
sb.destroy()

# Now, restore from the snapshot.
restored_sandbox = sandbox_manager.restore('my-agent-sandbox', snapshot_id)
output = restored_sandbox.run_cmd("cat /tmp/progress.txt")

# The output will be "Initial state"
print(output['output'])
restored_sandbox.destroy()
```

*   **GUI and Browser Access**: Each sandbox runs a VNC server, allowing access to a graphical user interface, which includes a pre-installed Chrome browser. Your agent can be programmed to perform tasks that require web interaction. The API provides the necessary port forwarding details to connect to the VNC server.

By combining these features, you can build an AI agent that can write code, execute it, debug it by inspecting files and processes, and even interact with web pages, all within a secure and isolated environment.

Formal verification for self-modifying systems uses rigorous mathematical proofs—often enforced as “proof obligations” and checked by theorem provers—to ensure that each proposed self-modification preserves essential properties such as correctness, safety, and intended utility. This approach, inspired by the Gödel machine architecture, prevents “improvements” that would break, regress, or destabilize the agent[1][2][3][4][5].

## Key Elements of Formal Verification for Self-Modification

### 1. Proof Obligations for Each Change
- Every proposed code or policy modification triggers a set of formal “proof obligations”: logical assertions about desired properties (correctness, safety, no regression, resource bounds, etc.) that must be proven true before the change can be approved or rolled out[1].
- Examples:
  - “The new policy must perform at least as well as the old one for all test cases.”
  - “No critical invariants (e.g., safety boundaries or liveness requirements) are violated by the new code.”

### 2. Dedicated Verification Agent
- Self-modifying agents are architected as two components:  
   - The “Improver” or “Self-Modifier” proposes and drafts modifications[1].
   - The “Verification Agent” or “Verifier” analyzes the change with formal methods (Coq, Lean, Z3 SMT solver, etc.), proving that the modification upholds all relevant properties[1][4].
- If the proof cannot be established, the change is rejected and rolled back automatically.

### 3. Safety and Meta-Invariant Checking
- The verification agent encodes systemic properties (e.g., “No unauthorized access”, “Ethical boundaries respected”) as invariants[1][5].
- Modifications are only accepted if they provably preserve these meta-invariants, ensuring the system doesn’t “self-improve” into unsafe or nonfunctional states.

### 4. Iterative, Layered Verification
- Upon complex or critical changes, verification is carried out in layers:
  - First, low-level (syntactic) checks (e.g., passes all tests, no runtime errors).
  - Then, semantic and behavioral proofs about policy and performance.
  - Finally, higher-level meta-properties—such as ensuring the overall improvement loop itself cannot deadlock or violate value-alignment[1][5].

### 5. Tooling and Automation
- Formal verification tools like Coq or Lean (proof assistants) and Z3 (SMT solver) automate the verification, reduce manual proof labor, and ensure changes are correct “by construction”[1][6][7].
- Proofs, verification outcomes, and all code deltas are logged for full auditability and reproducibility.
